Real-Time Chat Application

The One Concept Everything Hangs On

A chat application is fundamentally a message routing system. A user sends a message addressed to a conversation (which could be one other person or a group). The system's job is to store that message durably so it is never lost, and then deliver it to every participant as quickly as possible — ideally in real time via a persistent connection, but falling back to a pull-based approach when real-time delivery fails. Everything else — read receipts, presence indicators, push notifications — layers on top of this core store-and-forward loop.

Assumptions

  • This is a consumer-facing chat app similar to WhatsApp or Slack, supporting 1-on-1 and group conversations
  • Users authenticate via email/password or OAuth; authentication system itself is out of scope
  • The app must support text messages initially; file/image attachments are a later extension
  • Target scale is 50 million monthly active users with up to 5 million concurrent connections
  • Messages must be delivered in real time (under 500ms end-to-end) when both users are online
  • Messages must be stored persistently so users can see history when they come back online
  • The system must handle users on unreliable mobile networks gracefully

Requirements and Scope

Functional Requirements

  • Users can send and receive text messages in real time
  • Users can create 1-on-1 conversations and group conversations (up to 500 members)
  • Messages are persisted and available as scrollable history
  • Users see a conversation list sorted by most recent activity
  • Messages show delivery status: sent, delivered, read
  • Users receive push notifications for messages received while the app is backgrounded
  • Users can see which contacts are currently online (presence)
  • Messages are delivered in the order they were sent within a conversation

Non-Functional Requirements

  • Real-time delivery latency under 500ms p99 for online recipients
  • Message durability: zero message loss once the server acknowledges a send
  • 99.99% availability for message send and receive
  • Support 5 million concurrent WebSocket connections
  • Message history loads in under 200ms for recent messages
  • End-to-end encryption for 1-on-1 conversations

Out of Scope

  • Voice and video calling
  • File and image attachments (designed for but not built in v1)
  • Message search across all conversations
  • Administrative moderation tools
  • User registration and authentication (assumed to exist)

Scale and Constraints

  • Read/Write Ratio: 5:1 — Each message is written once but read by 1+ recipients, plus history fetches. Group messages amplify reads.
  • Request Rate: Writes: ~23K messages/sec (50M MAU × 40 msgs/day ÷ 86400 sec, with peak 2× average). Reads: ~115K/sec including real-time pushes and history fetches.
  • Storage Growth: ~1.5 TB/year for message text alone (50M users × 40 msgs × 200 bytes × 365 days). With metadata, indexes, and replication: ~10 TB/year.
  • Latency: p50 < 100ms, p99 < 500ms for message delivery to online recipients. History fetch p99 < 200ms.
  • Availability: 99.99% (about 52 minutes of downtime per year). Messages must never be silently lost.

API Contracts

Send Message

POST /v1/conversations/{conversationId}/messages

Sends a text message to a conversation. The server assigns a server-side timestamp and message ID, stores the message, and pushes it to online recipients via their WebSocket connections. Returns immediately after durable storage — delivery to recipients happens asynchronously.

Request

{
  "clientMessageId": "cli_8f14e45f-ceea-367f-a27f-4eb04ef0e7db",
  "text": "Hey, are we still meeting at 3?",
  "replyTo": null
}

Response

{
  "messageId": "msg_20260706T143022_a7b3c",
  "conversationId": "conv_xk92m",
  "clientMessageId": "cli_8f14e45f-ceea-367f-a27f-4eb04ef0e7db",
  "senderId": "usr_alice",
  "text": "Hey, are we still meeting at 3?",
  "sentAt": "2026-07-06T14:30:22.481Z",
  "status": "sent"
}

Fetch Message History

GET /v1/conversations/{conversationId}/messages?before={cursor}&limit=50

Returns a page of messages older than the given cursor, sorted newest-first. Used for scrolling back through conversation history. The cursor is an opaque string (typically a server timestamp + message ID) that the client passes from the previous response.

Request

GET /v1/conversations/conv_xk92m/messages?before=msg_20260706T143022_a7b3c&limit=50

Response

WebSocket Connection

GET /v1/ws?token={authToken}

Upgrades to a WebSocket connection for real-time message delivery. Once connected, the server pushes new messages, delivery receipts, read receipts, typing indicators, and presence updates. The client sends typing indicators and read-receipt acknowledgments over this connection.

Request

WebSocket frames from server:
{"type": "message", "data": {"conversationId": "conv_xk92m", "messageId": "msg_...", "senderId": "usr_alice", "text": "Hello!", "sentAt": "..."}}
{"type": "receipt", "data": {"conversationId": "conv_xk92m", "messageId": "msg_...", "status": "delivered", "userId": "usr_bob"}}
{"type": "typing", "data": {"conversationId": "conv_xk92m", "userId": "usr_bob", "isTyping": true}}

Response

WebSocket frames from client:
{"type": "ack", "messageId": "msg_..."}
{"type": "read", "conversationId": "conv_xk92m", "upTo": "msg_..."}
{"type": "typing", "conversationId": "conv_xk92m", "isTyping": true}

Update Read Position

POST /v1/conversations/{conversationId}/read

Marks all messages up to a given message ID as read by the current user. This updates the unread count for the conversation list and triggers read receipts to the message senders.

Request

{
  "lastReadMessageId": "msg_20260706T143022_a7b3c"
}

Response

{
  "conversationId": "conv_xk92m",
  "lastReadMessageId": "msg_20260706T143022_a7b3c",
  "unreadCount": 0

High-Level Architecture

High-Level Architecture

The system has three main server-side components: API servers handle REST requests (send message, fetch history, manage conversations), WebSocket gateways maintain persistent connections for real-time delivery, and background workers process push notifications and analytics. Redis acts as both a cache and a pub/sub bus that connects API servers to WebSocket gateways. Cassandra stores messages (optimized for write-heavy, time-series-like access), while PostgreSQL stores users and conversation metadata (relational queries like 'list my conversations sorted by recent activity').

flowchart LR Client[Mobile / Web Client] -->|WebSocket| LB[Load Balancer] Client -->|HTTPS REST| LB LB --> WS[WebSocket Gateway\nCluster] LB --> API[API Servers] API --> DB[(Message DB\nCassandra)] API --> UserDB[(User & Conversation DB\nPostgreSQL)] API --> Cache[(Redis Cache\n& Pub/Sub)] WS --> Cache API --> MQ[Message Queue\nKafka] MQ --> Notify[Push Notification\nWorker] MQ --> Analytics[Analytics\nPipeline] Notify --> APNS[APNs / FCM]

Message Send Flow

When Alice sends a message: (1) the API server writes it to the message database, (2) acknowledges to Alice with the server-assigned ID and timestamp, (3) publishes a notification to Redis pub/sub on the conversation's channel, (4) the WebSocket gateway subscribed to that channel pushes the message to Bob's open connection, (5) Bob's client acknowledges receipt, which triggers a delivery receipt back to Alice, (6) simultaneously, a notification job is enqueued to Kafka so that if Bob is offline, a push notification is sent via APNs or FCM.

sequenceDiagram participant Alice as Alice's Client participant API as API Server participant DB as Message DB participant Redis as Redis Pub/Sub participant WS as WebSocket Gateway participant Bob as Bob's Client participant Queue as Kafka participant Push as Push Worker Alice->>API: POST /messages (text) API->>DB: Write message DB-->>API: Ack API-->>Alice: 200 OK (messageId, sentAt) API->>Redis: PUBLISH conv_xk92m Redis->>WS: Notification: new message WS->>Bob: WebSocket frame (message) Bob-->>WS: Ack WS->>API: Delivery receipt API->>Redis: PUBLISH receipt Redis->>WS: Receipt for Alice WS->>Alice: WebSocket frame (delivered) API->>Queue: Enqueue notification job Queue->>Push: If Bob offline Push->>Bob: Push notification (APNs/FCM)

Data Partitioning Strategy

Messages are partitioned by conversationId in Cassandra, giving each conversation its own sorted timeline. PostgreSQL holds relational data (users, conversations, membership) where JOIN queries are needed. Redis serves three roles: pub/sub channels for real-time fan-out, a session cache mapping users to their WebSocket gateway node, and fast unread count tracking.

High-Level Architecture

The chat system is built around three server-side layers that separate concerns cleanly: **1. API Servers (stateless HTTP handlers)** These handle all REST API requests: sending messages, creating conversations, fetching history, and updating read positions. They are stateless — any API server can handle any request — which makes horizontal scaling straightforward. When a message is sent, the API server writes it to the message database, publishes a notification to Redis pub/sub, and enqueues a push-notification job to Kafka. **2. WebSocket Gateway (stateful connection holders)** These servers maintain long-lived WebSocket connections with clients. Each gateway server subscribes to Redis pub/sub channels for all conversations that its connected users belong to. When a notification arrives on a channel, the gateway forwards the message to the appropriate WebSocket connection. The gateway cluster is the most operationally complex component because it holds state (the connections themselves). A session registry in Redis maps each online userId to the gateway node holding their connection, so API servers know which pub/sub channel to notify. **3. Background Workers** Kafka consumers that handle push notifications (for offline users), analytics event processing, and any heavy async work like media processing (future). These are fully decoupled from the request path. **Request Flow for Sending a Message:** 1. Client sends POST /v1/conversations/{id}/messages to an API server via the load balancer. 2. API server validates the request and checks conversation membership. 3. API server writes the message to Cassandra (message store) and updates lastMessageAt in PostgreSQL. 4. API server returns 200 with the message ID and server timestamp to the sender. 5. API server publishes the message to the Redis pub/sub channel for that conversation. 6. Every WebSocket gateway subscribed to that channel receives the notification. 7. Each gateway checks which of its connected users are members of that conversation and pushes the message over their WebSocket. 8. For members who are NOT online (no active WebSocket), the API server enqueues a push notification job to Kafka. 9. A push notification worker dequeues the job and sends it via APNs (Apple) or FCM (Google). **Why two databases?** Messages and conversations have fundamentally different access patterns. Messages are append-heavy (write a lot, read sequentially within a conversation) — Cassandra excels at this because its partition-key design gives each conversation a sorted log of messages. Conversations and memberships require relational queries (list my conversations with unread counts, check membership) — PostgreSQL handles these well with indexes and JOINs. **Why Redis sits in the middle?** Redis pub/sub provides the glue between stateless API servers and stateful WebSocket gateways. When an API server writes a message, it does not need to know which gateway holds the recipient's connection. It just publishes to the conversation's channel, and the right gateway picks it up. This decoupling means API servers and gateways scale independently.

Deep Dives

Real-Time Message Delivery via WebSocket

**What problem is this solving?** Users expect chat messages to appear instantly — within a fraction of a second — without refreshing or polling. HTTP polling would either waste bandwidth (polling every second) or introduce visible delay (polling every 10 seconds). We need a persistent, bidirectional channel between client and server. **Simplest version:** Each client opens a WebSocket connection to a gateway server. The gateway maintains an in-memory map of userId → WebSocket connection. When a message arrives for a conversation, the gateway looks up all online members and pushes the message frame directly. **What breaks at scale?** With 5 million concurrent connections, a single gateway server cannot hold them all (each connection uses ~20 KB of memory, so 5M connections = 100 GB). We need a cluster of gateway servers. But now the problem is: when API Server A wants to deliver a message to user Bob, how does it know which gateway server holds Bob's connection? **What do we choose?** We use Redis pub/sub as the coordination layer. Each gateway server subscribes to pub/sub channels for every conversation its connected users belong to. When a user connects, the gateway subscribes to all their conversation channels. When a message is sent, the API server publishes to the conversation's channel, and only the gateway(s) with relevant users receive it. Additionally, we maintain a session registry in Redis: a simple key-value map of userId → gatewayNodeId. This is used for targeted delivery (e.g., typing indicators to a specific user) without broadcasting. **Connection lifecycle:** 1. Client connects to /v1/ws with an auth token. 2. Load balancer routes the connection to a gateway (using sticky sessions by userId hash). 3. Gateway authenticates the token, registers the connection in Redis, and subscribes to all conversation channels for this user. 4. Gateway sends a heartbeat ping every 30 seconds. If no pong within 10 seconds, the connection is closed and the Redis session entry is removed. 5. On disconnect, the gateway unsubscribes from channels and removes the session entry. **What tradeoff are we accepting?** Redis pub/sub is fire-and-forget — if a gateway misses a message (e.g., during a brief network partition between gateway and Redis), the message is not re-delivered via pub/sub. This is acceptable because the message IS durably stored in Cassandra. When a client reconnects, it fetches missed messages via the REST API using its last-known messageId as a cursor. Real-time delivery is best-effort; persistence is guaranteed.

Message Storage and Partitioning in Cassandra

**What problem is this solving?** We need to store billions of messages and retrieve them efficiently per conversation, sorted by time, with fast writes and fast range reads. **Simplest version:** A single PostgreSQL table with (conversationId, sentAt, messageId, text, senderId). Index on (conversationId, sentAt DESC). This works at small scale. **What breaks at scale?** At 23K writes/sec and growing storage, a single relational database becomes a bottleneck. Sharding PostgreSQL by conversationId is possible but operationally complex. More importantly, the access pattern — append-only writes followed by sequential reads within a partition — is exactly what wide-column stores like Cassandra are designed for. **What do we choose?** Apache Cassandra with this table design: ``` Primary key: (conversationId, sentAt, messageId) Partition key: conversationId Clustering columns: sentAt DESC, messageId DESC ``` This means all messages for a conversation are stored together on the same set of nodes, sorted by timestamp. Fetching the latest 50 messages is a single partition read with a LIMIT clause — extremely fast. **Partition sizing concern:** A very active conversation (e.g., a group with years of history) could grow its partition to hundreds of MB. Cassandra partitions should ideally stay under 100 MB. Solution: use a compound partition key of (conversationId, timeBucket) where timeBucket is a monthly bucket like '2026-07'. When fetching history, the API server queries the current bucket first, and only reads older buckets if the user scrolls further back. Most queries never go past the current bucket. **Write path:** Writes use consistency level LOCAL_QUORUM (at least 2 of 3 replicas in the local datacenter must acknowledge). This ensures the message is durable before we ack to the client. Cassandra's log-structured storage makes writes very fast — typically under 5ms. **What tradeoff are we accepting?** Cassandra does not support ad-hoc queries well (no full-text search, no JOINs). We accept this because messages have a narrow access pattern. If we need message search later, we will build a separate search index (Elasticsearch) fed by a Kafka consumer — not by querying Cassandra with WHERE text LIKE '%...'.

Presence and Typing Indicators

**What problem is this solving?** Users want to know if the person they're chatting with is online (presence) and whether they're currently typing. These are ephemeral signals — they don't need to be stored persistently, but they need to be delivered quickly. **Simplest version:** Presence: When a user's WebSocket connects, mark them as 'online' in Redis with a TTL of 60 seconds. The gateway refreshes this key every 30 seconds (on each heartbeat). When the key expires, the user is considered offline. To check if someone is online, just look up their key in Redis. Typing: When a user starts typing, their client sends a {type: 'typing', isTyping: true} frame over the WebSocket. The gateway publishes a typing event to the Redis pub/sub channel for that conversation. Other gateways push it to online members. Typing events have a short TTL on the client side (3 seconds) — if no new typing event arrives, the 'typing' indicator disappears. **What breaks at scale?** Presence at scale is expensive if you try to track it precisely. If 5 million users are online and each user has 100 contacts, broadcasting 'user X went offline' to 100 people on every disconnect creates a storm of updates. This is the classic presence fan-out problem. **What do we choose?** Lazy presence: Don't broadcast presence changes proactively. Instead, when a user opens a conversation, their client queries the presence status of the other participants via a REST endpoint (GET /v1/users/{id}/presence). The response comes from Redis (a simple key lookup, very fast). The WebSocket gateway sends presence updates only for users in conversations that the current user has open — not for their entire contact list. For typing indicators, the scope is already narrow (only sent within a conversation while it's being viewed), so the pub/sub approach works without modification. **What tradeoff are we accepting?** Presence status may be stale by up to 60 seconds (the TTL window). A user who closed the app might still appear 'online' briefly. This is acceptable — all major chat apps have this same behavior, and users understand that presence is approximate.

Push Notifications for Offline Users

**What problem is this solving?** When a recipient is not connected via WebSocket, they need to be notified about new messages via mobile push notifications (APNs for iOS, FCM for Android). This must be reliable but cannot block the message-send path. **Simplest version:** After writing a message, the API server checks Redis for the recipient's session entry. If no active session exists, it enqueues a push notification job to Kafka with the message preview and the recipient's userId. A push notification worker consumes from Kafka, looks up the user's push token from PostgreSQL (or a Redis cache), and calls the APNs/FCM API to deliver the notification. **What breaks at scale?** Group messages create amplification: a message to a 200-person group generates up to 200 push notification checks. If 150 of those users are offline, that's 150 push API calls. At 23K messages/sec with group amplification, the push worker cluster must handle hundreds of thousands of push calls per second. **What do we choose?** Batch and throttle push notifications: 1. **Batching:** Group multiple notifications for the same user within a short window (2 seconds) into a single push. If Alice receives 5 messages in 2 seconds, she gets one push saying '5 new messages' rather than 5 separate pushes. 2. **Suppression:** Don't send a push if the user's WebSocket connection is active (they'll get the message in real time). Don't send a push if the user has the conversation muted. 3. **Rate limiting:** Cap push notifications at 1 per conversation per 30 seconds per user to avoid spamming. 4. **Separate Kafka partitions:** Partition the notification topic by userId so all notifications for a single user go to the same consumer, enabling the batching logic. **What tradeoff are we accepting?** The 2-second batching window means push notifications are slightly delayed compared to real-time WebSocket delivery. This is imperceptible to users and dramatically reduces push API costs and device battery drain.

Conversation List and Unread Counts

**What problem is this solving?** The conversation list is the most-viewed screen in any chat app. It must load fast, show conversations sorted by recent activity, and display accurate unread counts. This is a read-heavy access pattern that touches multiple tables. **Simplest version:** Query PostgreSQL: SELECT conversations with their lastMessageAt and lastMessagePreview, JOIN with ConversationMembers to filter by the current user, compute unread count by comparing the user's lastReadMessageId with the conversation's most recent message. Sort by lastMessageAt DESC. **What breaks at scale?** Computing unread counts on every list load requires comparing timestamps or counting messages per conversation. If a user belongs to 200 conversations, this becomes expensive at request time. The conversation list is also the first thing users see when they open the app — it must load in under 100ms. **What do we choose?** Maintain pre-computed unread counts in Redis: - Key: `unread:{userId}:{conversationId}` → integer count - When a new message arrives in a conversation, increment the unread counter for every member except the sender. - When a user reads a conversation (POST /conversations/{id}/read), reset their counter to 0. - When loading the conversation list, fetch conversation metadata from PostgreSQL (or a cache) and unread counts from Redis in a single MGET call. This turns the expensive per-request computation into cheap increment/reset operations that happen at write time. **Why not compute from message data?** Counting messages in Cassandra between two timestamps is a full-partition scan — expensive and slow. Pre-computing in Redis trades a small amount of write-path complexity for dramatically faster reads. **What tradeoff are we accepting?** The Redis unread count could drift if a Redis write fails (e.g., during a brief network issue). We mitigate this with a periodic reconciliation job that recalculates counts from the source of truth (Cassandra + PostgreSQL) and corrects any drift. A slightly wrong unread count for a few minutes is acceptable; losing messages is not.

End-to-End Encryption for Direct Messages

**What problem is this solving?** Users expect that their private 1-on-1 conversations cannot be read by the chat service operator or anyone who gains access to the server-side database. End-to-end encryption (E2EE) means only the sender and recipient can decrypt messages — the server sees only ciphertext. **Simplest version:** Use the Signal Protocol (the same protocol used by Signal, WhatsApp, and Google Messages). Each user generates a long-term identity key pair and a set of one-time pre-keys on their device. These public keys are uploaded to the server. When Alice wants to message Bob for the first time, she fetches Bob's public pre-keys from the server and uses them to establish a shared session. All subsequent messages are encrypted with session keys derived from this exchange. **Key concepts (plain English):** - **Identity key:** A permanent public/private key pair tied to a user's device. The private key never leaves the device. - **Pre-keys:** Temporary public keys uploaded to the server. They allow someone to start an encrypted session even if the recipient is offline. - **Double Ratchet:** An algorithm that generates a new encryption key for every single message. Even if one key is compromised, past and future messages remain secure. This property is called 'forward secrecy.' **What breaks at scale?** E2EE for group messages is much harder. The Signal Protocol works for 1-on-1 easily, but for a group of N people, naive pairwise encryption means each message must be encrypted N times. The Sender Keys extension allows a sender to share a symmetric key with all group members, so they encrypt only once — but this weakens forward secrecy because a compromised sender key decrypts all messages until the key is rotated. **What do we choose?** Implement E2EE for 1-on-1 conversations in v1 using the Signal Protocol (via the open-source libsignal library). Group E2EE is deferred to v2, where we'll evaluate the MLS (Messaging Layer Security) standard, which is designed for efficient group encryption. **What tradeoff are we accepting?** With E2EE, the server cannot index or search message content. Features like server-side search, link previews generated server-side, and spam detection become impossible for encrypted conversations. These features must either work client-side or be limited to unencrypted group conversations.

Once consumers catch up, all pending notifications are delivered (with batching to avoid spamming).Auto-scaling consumer groups: if lag exceeds a threshold, Kubernetes spins up additional consumer pods. Alert on consumer lag > 30 seconds. Kafka topic retention is set to 72 hours, providing ample buffer for recovery.
Client sends duplicate message (network retry)The client retries a message send after a timeout. The API server checks the clientMessageId against recently stored messages for the conversation. If a match is found, it returns the existing message with its server-assigned ID instead of creating a duplicate. The user sees only one message.Idempotency is enforced at the API layer using the client-generated clientMessageId. A secondary index on (conversationId, clientMessageId) in Cassandra makes the dedup check fast. The dedup window is 24 hours — long enough to cover any realistic retry scenario.
User database (PostgreSQL) unavailableConversation list and membership queries fail. Sending messages to existing conversations still works (membership can be cached in Redis for the short term). Creating new conversations fails. User profile lookups fall back to a Redis cache (stale but functional).PostgreSQL runs as a primary with synchronous replicas in two availability zones. Automatic failover promotes a replica within 30 seconds. Read replicas serve read traffic during the failover window.
Unread count drift in RedisA user's unread count for a conversation is slightly wrong — showing 3 instead of 4, or 1 instead of 0. The conversation list may display an inaccurate badge count. No messages are lost; only the counter is wrong.A reconciliation job runs every 5 minutes, recalculating unread counts from the source of truth (the user's lastReadMessageId vs. messages in Cassandra) and correcting Redis. When a user opens a conversation, the count is also recalculated on the fly.

Tradeoffs

DecisionCostBenefit
Use Cassandra for messages and PostgreSQL for users/conversations (polyglot persistence)Operational complexity of running two database systems. Engineers must understand two query languages and two consistency models. Cross-store queries (e.g., 'messages from user X') require application-level joins.Each database is used for what it does best. Cassandra handles high-throughput, append-only message writes with partition-based reads. PostgreSQL handles relational membership queries and transactional updates. Neither is forced into an access pattern it's bad at.
Redis pub/sub for real-time delivery instead of direct gateway-to-gateway communicationRedis pub/sub is fire-and-forget — no delivery guarantee. If a message is published while a gateway is briefly disconnected from Redis, that message is not re-delivered via pub/sub. Adds a dependency on Redis availability for real-time features.Complete decoupling between API servers and WebSocket gateways. Neither needs to know the other's topology. Adding or removing gateway nodes requires zero configuration changes. Massively simplifies horizontal scaling.
Pre-computed unread counts in Redis instead of counting at read timeCounts can drift if Redis writes fail. Requires a reconciliation job. Slightly more write-path complexity (increment counter on every message, reset on read).Conversation list loads in under 50ms instead of hundreds of milliseconds. The conversation list is the single most frequently loaded screen — optimizing it has outsized impact on perceived performance.
Lazy presence (query on demand) instead of proactive presence broadcastingPresence status can be stale by up to 60 seconds. Users may see someone as 'online' briefly after they've actually closed the app.Eliminates the presence fan-out storm. Broadcasting 'user X went offline' to 200 contacts of every disconnecting user would create unsustainable event volume at 5M concurrent users. Lazy presence scales linearly with active conversations, not with contact list size.
End-to-end encryption for 1-on-1 only (not groups in v1)Group conversations are not encrypted in v1, which may be a concern for privacy-focused users. Requires communicating this limitation clearly.Dramatically simpler implementation. 1-on-1 E2EE with Signal Protocol is well-understood and well-tested. Group E2EE (MLS) is still maturing as a standard. Shipping 1-on-1 E2EE quickly gives users real privacy protection where it matters most.
Kafka for async push notification processing instead of synchronous push from the API serverPush notifications are delayed by 1-3 seconds compared to real-time delivery. Adds Kafka as an infrastructure dependency.The message-send API response time is not affected by slow push provider APIs (APNs can take 200-500ms). Push notification failures don't affect message delivery. Enables batching (reducing push spam) and rate limiting. Push workers scale independently from API servers.
Time-bucketed Cassandra partition keys (conversationId + monthBucket)Queries that cross month boundaries require reading from two partitions. Client-side logic must handle the bucket boundary. Slightly more complex query code.Prevents any single partition from growing unboundedly large. A very active group chat that has been running for years won't degrade in read performance because each month's messages are in a separate, reasonably-sized partition.
WebSocket heartbeat with 30-second intervalA crashed client may appear 'online' for up to 40 seconds (30s heartbeat + 10s timeout) before being detected as disconnected. Minor network overhead from heartbeat pings.Reliable detection of dead connections without relying on TCP keepalive (which varies by OS and is often set to 2+ hours). 30 seconds strikes a balance between quick detection and low overhead — shorter intervals would significantly increase per-connection bandwidth usage at 5M connections.

Glossary

WebSocket
A communication protocol that provides a persistent, bidirectional channel between a client and server over a single TCP connection. Unlike HTTP (where the client must ask for data each time), a WebSocket lets the server push data to the client at any time, enabling real-time features like instant message delivery.
Pub/Sub (Publish/Subscribe)
A messaging pattern where senders (publishers) broadcast messages to a channel without knowing who will receive them, and receivers (subscribers) listen to channels they care about without knowing who sends to them. Redis pub/sub is used here so that API servers can publish 'new message' events without knowing which WebSocket gateway holds the recipient's connection.
Cassandra
An open-source distributed database designed for handling large amounts of write-heavy data across many servers. Data is organized into partitions (groups of rows stored together), and each partition is replicated across multiple nodes for durability. It's ideal for chat messages because it writes fast and reads efficiently within a partition.
Partition Key
In Cassandra, the field(s) that determine which node stores a particular row of data. All rows with the same partition key are stored together on the same set of nodes, which makes reading them very fast. In this design, conversationId is the partition key for messages, so all messages in a conversation are co-located.
LOCAL_QUORUM
A Cassandra consistency level that requires a majority of replicas in the local datacenter to acknowledge a write before returning success to the client. With a replication factor of 3, this means 2 out of 3 replicas must confirm the write. This provides strong durability without the latency cost of waiting for acknowledgments from remote datacenters.
Kafka
A distributed event streaming platform used as a message queue. Producers write messages (events) to topics, and consumers read from those topics. Messages are stored durably and can be replayed. In this design, Kafka decouples the message-send path from push notification delivery, allowing each to scale independently.
APNs (Apple Push Notification service)
Apple's cloud service for sending push notifications to iOS and macOS devices. The chat server sends a notification payload to APNs, which routes it to the user's device even when the chat app is not actively running.
FCM (Firebase Cloud Messaging)
Google's cloud service for sending push notifications to Android devices (and optionally iOS and web). Similar to APNs but for the Android ecosystem.
Signal Protocol
An encryption protocol that provides end-to-end encryption for messaging. It uses a combination of long-term identity keys and ephemeral session keys to ensure that only the sender and recipient can read messages. The protocol also provides 'forward secrecy,' meaning that if a key is compromised, past messages remain secure.
Double Ratchet
A key management algorithm used in the Signal Protocol that generates a new unique encryption key for every single message. It's called a 'ratchet' because it only moves forward — you can derive future keys from current ones, but not past keys. This ensures that compromising one message's key doesn't reveal any other messages.
Forward Secrecy
A property of an encryption system where compromising a current key does not allow decryption of past messages. Each message is encrypted with a different key, and old keys are deleted after use. Even if an attacker obtains your current key, they cannot read your message history.
Idempotency
The property of an operation where performing it multiple times has the same effect as performing it once. In this design, sending a message with the same clientMessageId twice results in only one stored message. This protects against duplicate messages when clients retry after network timeouts.
Fan-out
The process of distributing a single event to multiple recipients. When a message is sent to a group conversation with 100 members, it must be 'fanned out' to up to 100 WebSocket connections and potentially 100 push notification calls. Fan-out is one of the primary scaling challenges in chat systems.
Sticky Session
A load balancing strategy where requests from the same client are always routed to the same server. For WebSocket connections, the load balancer uses a hash of the userId to ensure a user's connection always goes to the same gateway node, simplifying session management.
MLS (Messaging Layer Security)
An emerging IETF standard for efficient group end-to-end encryption. Unlike encrypting a message separately for each group member (which is expensive for large groups), MLS uses a tree-based key structure that allows efficient key updates as members join or leave. It's the likely successor to current group E2EE approaches.
Polyglot Persistence
An architecture pattern where different types of data are stored in different database systems, each chosen for its strengths. In this design, messages go to Cassandra (optimized for write-heavy, time-series data) while users and conversations go to PostgreSQL (optimized for relational queries and transactions).
